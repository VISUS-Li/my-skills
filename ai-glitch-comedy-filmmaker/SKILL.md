# AI Glitch Comedy Filmmaker

## 核心定义

创作这种内容：

> 人类用很正常、很具体的语言要求 AI 制作一个粗糙但可玩的 3D 游戏功能；AI 或编程 Agent 表面宣布完成，实际给出一个在画面上明显错误、在技术上勉强说得通的结果；玩家继续测试、利用和放大错误；旁白用冷静、精确、略带无奈的方式重新定义事故；错误最后改变玩法、胜负或系统判定。

核心不是完整剧情，也不是随机 BUG 合集。**剧情可以没有，但成片旁白不能只是点缀。** 对视频类输出，旁白承担目标说明、预期建立、错误重定义、测试观察、修复转场和结尾回收，是连接粗糙游戏画面的主叙述轨。核心是三个同时成立的层：

1. **正常承诺**：观众马上知道本来要做什么。
2. **3D 视觉反差**：实际结果在一个镜头里就能看懂，而且比文字更好笑。
3. **吐槽式重新定义**：旁白不大喊“离谱”，而是用一本正经的技术逻辑为错误命名。

标准笑点示例：

- 开场画面：漂浮勇者穿墙，恶龙向他低头。
- 恶龙对白：“冥界的同事终于来了。”
- 回到提示词界面，旁白：“事情原本没有这么复杂。我只让 AI 做一个勇者救公主的游戏。”
- 请求：给勇者添加自然跑步动画。
- AI 声明：已添加自然跑步动画。
- 实际画面：角色四肢脱离身体，身体悬浮前进。
- 留白：画面保持 0.8 秒，只听见木块碰撞。
- 旁白：自然，主要是指不受人体结构约束。
- 连续旁白：“我原本以为这只是动画问题。直到他的身体进了城，碰撞体还在门口接受检查。”
- 玩法后果：角色的碰撞体仍在原地，身体模型可以穿墙侦察。

只吸收可迁移的无厘头短喜剧机制：小人物、类型降格、字面误解、低成本自觉、冷面吐槽、权威合理化、快速反转和结尾回收。不要复制特定创作者的精确台词、固定人物、口头禅或现成桥段。

## 永久优先级

始终按以下顺序决策：

1. 3D 画面能否一眼看懂。
2. 玩家能否操作、触发或利用事故。
3. Agent 能否在粗糙原型里稳定实现。
4. 笑点是否由“承诺—画面—冷句”构成。
5. 错误是否会进入另一个游戏系统并升级。
6. 成片是否有连续、完整、可直接录音的旁白主线。
7. 视频是否有清楚的镜头、声音和剪辑节拍。
8. 剧情是否真的需要；默认不需要。

禁止先写完整故事再倒推游戏。禁止要求 Agent 构建完整商业游戏。禁止用复杂美术、开放世界、大量 NPC 或长剧情掩盖一个不清楚的核心笑点。

## 默认条件

用户未指定时，使用：

- 维度：`3d`。
- 原型精度：`rough_graybox`。
- 镜头：第三人称跟随 + 可切换自由摄像机。
- 场景：1 个小型灰盒场景，30–60 秒内可走完。
- 角色：1 个玩家，0–3 个 NPC。
- 资产：基础几何体、默认材质、低模或现有占位角色；不依赖新购买资产。
- 系统：玩家控制 + 1 个主失控系统 + 1 个耦合系统 + 胜负/UI。
- 内容：3–6 分钟横屏主片，附 30–60 秒竖屏切片。
- 旁白：第一人称创作者/玩家视角；每成片分钟 120–180 个汉字；至少 70% 的观众分镜含旁白或对白；除刻意视觉留白外不得连续无声超过 12 秒。
- 故事：`story.enabled = false`，除非角色关系明显增强事故。剧情可选，旁白主线必选。
- Agent 能力假设：理解和执行能力有限，容易过度修改、遗漏场景配置、擅自优化或把故意错误修掉。

## 选择任务模式

1. **3D 游戏概念**：生成玩家动词、场景、系统和主要视觉事故。
2. **Agent 实现包**：为 Cursor 等生成可逐项执行的高约束任务包。
3. **失控实验**：设计提示词、预期结果、错误空间和参数测试。
4. **粗糙原型**：定义最小场景、脚本、组件、调试开关和验收标准。
5. **无剧情视频**：围绕测试、试玩、挑战或伪产品演示组织内容。
6. **叙事包装**：只在玩法成立后增加少量身份、关系、旁白和对白。
7. **事故诊断**：分析真实 BUG，决定保留、放大、机制化、当背景笑点或修复。
8. **研究**：搜索平台案例和形式参考；区分直接同类、邻近形式和可迁移技法。

## 游戏优先工作流

### 1. 定义可玩的粗糙 3D 原型

先写清：

- 玩家幻想与玩家动词；
- 30–60 秒核心循环；
- 正常规则、胜利和失败；
- 场景尺寸、重要地标和动作路线；
- 主角、NPC、目标物的轮廓和相对尺度；
- 正常状态与失控状态在画面上的区别；
- 不做什么。

若不看旁白仍无法辨认“哪里错了”，必须先重做视觉设计。

### 2. 选择一个主失控系统

优先选择适合 3D、低成本且可视化强的系统：

- 动画与根运动脱钩；
- 骨骼、肢体、挂点或模型层级错误；
- 碰撞体与可见模型错位；
- 重力、摩擦、弹性、质量或约束极端；
- NPC 目标、阵营、跟随和 NavMesh 错误；
- 自动瞄准锁定错误对象；
- 任务区域、最近目标和胜负判断错误；
- 摄像机跟错对象或观察点；
- 物体复制、缩放或生成位置错误；
- 音效和动作错配。

主事故必须在一个全景或中景里可见，且玩家能主动触发、重复或利用。

### 3. 构造“承诺—反差—吐槽”单元

每个主要笑点必须填写：

1. `normal_request`：人类的正常需求。
2. `ai_claim`：AI 一本正经的完成声明。
3. `audience_expectation`：观众预期看到什么。
4. `visual_contrast`：实际 3D 画面，必须具体到身体、位置、速度、碰撞或对象关系。
5. `hold_sec`：错误揭示后的留白，默认 0.5–1.5 秒。
6. `deadpan_voiceover`：冷静重新定义，不提前解释画面。
7. `gameplay_consequence`：错误怎样改变操作或规则。
8. `escalation_link`：怎样进入下一次更大的错误。

旁白优先使用：

> 可验证事实 + 错误实现采用的狭窄逻辑 + 克制结论

例如：

> AI 没有让他跑得更自然。它只是把身体和四肢分别送到了终点。从团队协作的角度看，效率很高。

### 4. 设计玩法升级

至少三级，每一级改变游戏状态：

1. **视觉异常**：首先让观众笑，例如漂浮、穿墙、四肢分离。
2. **交互异常**：错误开始影响碰撞、敌人、任务或资源。
3. **规则接管**：玩家围绕错误制定策略，原胜负条件被替换或反向完成。
4. **可选回收**：系统正式宣布成功，同时展示最荒谬的代价。

不能只把同一 BUG 复制十次。数量增加只能作为视觉升级，不能替代系统变化。

### 5. 把设计拆成 Agent 能执行的任务

读取 [references/3d-agent-prompting.md](references/3d-agent-prompting.md)，并将工作拆成多个 `agent_task_packets`。一次只要求 Agent 完成一个可验证功能。

每个任务包必须包括：

- 当前项目事实和假设；
- 唯一目标；
- 目标文件和场景层级；
- 必需组件；
- 精确行为与参数；
- 画面目标示例；
- 实现步骤或伪代码；
- 允许修改和禁止修改；
- 非目标；
- 调试开关；
- 成功验收；
- 失败反例；
- Agent 回复格式。

禁止只写“做一个自然跑步动画”“优化 NPC”“增加搞笑 BUG”这种模糊指令。

### 6. 真实事故与拍摄版本分离

保留第一次真实事故作为开发素材；正式镜头使用可控版本：

- `chaos_enabled`；
- `chaos_level`；
- `fixed_seed`；
- `reset_scene`；
- `show_debug`；
- `freeze_ai`；
- `free_camera`；
- `normal_mode` 与 `shooting_mode`。

事故应在 10–30 秒内重现。Agent 无法稳定实现时，优先用简单脚本、动画层级错位、固定路径或预设参数重建，不追求真实复杂 BUG。

### 7. 组织视频

默认开发挑战结构：

**结果钩子 → 旁白交代“本来很简单” → 回到正常提示 → AI 完成声明 → 正常参照 → 错误揭示 → 0.5–1.5 秒留白 → 冷句重定义 → 旁白陪同玩家测试 → 修复请求 → 更大耦合 → 最终挑战 → UI 错误成功 → 最短回收句**

提示词界面和代码画面总计通常不超过主片 20%。视频主体必须是 3D 游戏画面与玩家操作。

### 8. 强制旁白主线

视频类交付（`video_blueprint`、`full_production`）必须同时满足：

- 输出从头到尾可直接录制的完整旁白稿，不得只写 `VO1`、`可选 VO`、`此处吐槽` 或摘要。
- 每成片分钟至少 120 个汉字，推荐 140–180 个；对白可补充但不能替代旁白主线。
- 至少 70% 的观众分镜包含实际旁白或对白。
- 每 15–25 秒至少出现一次叙述功能：目标、预期、观察、重定义、转场、修复动机、升级判断或回收。
- 除 0.5–1.5 秒视觉揭示留白、角色反应或结尾按钮外，不得出现超过 12 秒的无声区段。
- 每个主笑点至少有两句：揭示后的冷句，以及说明错误如何影响下一步玩法的后果句。
- 旁白必须是成片语言，不得出现 `chaos_level_1`、`debug target`、参数名或开发备注，除非这些词本身就是画面中的笑点。

旁白不是剧情。无剧情玩法实验仍需由创作者旁白串联“我要求什么—AI 做了什么—我怎样测试—系统如何进一步失控”。

### 9. 分离创作分镜与技术拍摄表

必须分别输出：

1. `video.creative_storyboard`：面向观众的成片分镜。写视觉、玩家操作、完整旁白/对白、留白、音乐、笑点转折。禁止用技术变量代替内容。
2. `video.technical_capture_plan`：面向开发和录制。写游戏状态、触发器、参数、调试开关、复位方式和机位坐标。

同一镜头用 `shot_id` 对齐，但不得混成一张技术日志式表格。

## 粗糙 3D 原型规则

允许：

- 方块墙体、平面地面、胶囊玩家；
- 默认天空和默认材质；
- 单一低模角色和重复 NPC；
- 简单 NavMesh、射线和触发器；
- 低精度动画和故意错位；
- 明显调试 UI；
- 一场景、一任务、一种敌人。

必须做到：

- 正常与错误版本轮廓差异明显；
- 摄像机能完整看到错误发生；
- 关键角色不被 UI、墙体或特效遮挡；
- 速度、距离、比例都有可调参数；
- 每次测试可复位；
- 关键错误不依赖偶发帧率或随机崩溃。

禁止：

- 在笑点未验证前做精细建模、复杂光照和完整关卡；
- 要求 Agent 自动寻找大量外部资产；
- 一次提示同时改控制器、动画、NPC、任务和 UI；
- 让 Agent“自由发挥”；
- 把卡死、闪退、无法操作当作合格喜剧；
- 让视觉错误太小，必须靠长旁白才能理解。

## 严格输出协议

所有创作类结果必须包含一个可解析 JSON 对象。默认顺序：

1. `project_spec` JSON；
2. 人类可读的创作说明；
3. Cursor/Agent 可直接复制的任务包；
4. 用户要求时给出完整分镜或脚本。

纯 JSON 模式：

- 只输出 JSON；
- 双引号、无注释、无尾逗号；
- 未知值用 `null`；
- 时间统一用秒；
- 枚举用英文 snake_case；
- 必须通过 `scripts/validate_output.py`。

顶层固定字段：

```json
{
  "schema_version": "4.0",
  "deliverable_type": "full_production",
  "language": "zh-CN",
  "project": {},
  "game": {},
  "ai_chaos": {},
  "comedy": {},
  "story": {"enabled": false},
  "video": {},
  "agent_execution": {},
  "implementation": {},
  "quality_check": {}
}
```

字段契约见 [references/format-contracts.md](references/format-contracts.md)，Schema 见 [references/output-schema.json](references/output-schema.json)。

视频类交付还必须读取 [references/narration-storyboard-contract.md](references/narration-storyboard-contract.md)。该文件定义旁白密度、完整台词、观众分镜与技术拍摄表的分离规则。

## 质量门槛

交付前检查：

- `game_first`：无剧情仍有可操作循环和目标。
- `three_dimensional_readability`：错误利用 3D 空间、身体、尺度、碰撞或镜头，并可一镜看懂。
- `rough_prototype_feasible`：可用占位资产和少量脚本完成。
- `player_agency`：玩家能触发、利用、对抗或放大事故。
- `visual_contrast`：正常期待和实际画面的差异具体且明显。
- `deadpan_voiceover`：单句吐槽克制、精确，不替观众喊“好笑”。
- `voiceover_density`：视频类输出达到最低字数和镜头覆盖率。
- `exact_voiceover_text`：所有旁白均为可直接录制的完整文本，无占位符和“可选”。
- `narrator_continuity`：旁白贯穿目标、测试、升级和回收，不只在一个笑点出现。
- `creative_technical_separation`：观众分镜不混入调试变量和开发日志。
- `promise_reveal_structure`：至少一个完整“请求—声明—画面—留白—冷句”单元。
- `systemic_escalation`：错误至少影响两个系统。
- `agent_specificity`：任务包包含参数、层级、约束、反例和验收。
- `reproducible`：可稳定触发和复位。
- `reversal`：胜负、规则、身份或目标至少一次反向。
- `callback`：有结尾型内容时回收早期元素。
- `scope_feasible`：不要求完整游戏或不必要美术。

任何项为 `false`，必须在 `quality_check.issues` 写明修复。

## 最小交付

即使用户只说“做一个 AI 失控的 3D 游戏视频”，也至少交付：

- 1 个粗糙 3D 场景规格；
- 2–4 个玩家动词；
- 1 个主失控系统；
- 1 个完整“承诺—反差—吐槽”笑点单元；
- 3 级玩法升级；
- 2–5 个可直接复制给 Cursor 的 Agent 任务包；
- 1 组调试与复位开关；
- 1 个录制钩子和分镜节拍；
- 视频类交付包含完整旁白稿，且不得出现 `VO1`、`可选旁白` 或空泛占位；
- 观众分镜与技术拍摄表各 1 份；
- 1 个通过验证的 JSON 对象。

不得只交付故事、段子、概念图或模糊开发建议。
