# 小白debug Reference Style DNA

This file is based on the uploaded reference video `小白debug1.mp4` plus direct frame inspection. Do not copy the creator's logo, copyrighted assets, exact edits, or voice identity. Use this as a style grammar for similar educational tech explainers.

## Measured reference facts

- File: `小白debug1.mp4`
- Duration: about 569.27s
- Resolution: 1920x1080
- FPS: 30
- Mode observed in sampled frames: desktop/tutorial explainer mixed with infographic cards, code/editor captures, browser UI, light-grid vector scenes, black terminal panels, and subtitle-first narration.

## Visual DNA

- **Canvas**: mostly 16:9 desktop/tutorial canvas. Light grid background appears in explanatory interludes; dark VS Code/terminal captures appear for technical proof.
- **Brand watermark**: small top-right creator mark. Do not copy; replace with user's own optional brand mark.
- **Caption system**: bottom-center Chinese subtitle pill, often short and punchy.
- **UI proof layer**: real software windows, browser pages, VS Code, terminal commands, settings panels, documentation pages.
- **Infographic layer**: rounded cards, arrows, small icons, comparison panels, question-mark cards, labels, simple vector illustrations.
- **Focus tools**: red arrows, glowing boxes, yellow highlight frames, enlarged code/command cards, check/error badges.
- **Scene variety**: alternates between live UI evidence and abstract explanation scenes, preventing a single visual mode from becoming stale.

## Motion DNA

- Use frequent attention shifts: highlight box, cursor arrow, zoom/punch-in, card slide, code panel enlarge, comparison split, quick diagram reveal.
- Keep the viewer's eye guided by one active focal cue at a time.
- For tutorial sections, make the UI action visible: button click, switch toggle, command line, file tree, code region, permission/error message.
- For explanation sections, convert ideas into a flow: file/card enters, module processes, output appears, stamp/check confirms.
- Use micro movement during holds: breathing zoom, highlight pulse, subtle background grid drift, cursor hover, blinking terminal caret.

## Audio DNA

- Voice remains dominant.
- Use small UI sounds for clicks/toggles, terminal hits, card snaps, error buzzes, and success chimes.
- Use short low hits for reveals or major warnings, but avoid constant whoosh spam.
- Use silence/drop or reduced bed before an important correction or “注意这里” moment.

- `robot_mascot`, `gear_module`, `pipe_arrow`, `switch_toggle`, `progress_bar`

Each asset must be treated as an actor with timed behavior in `asset_choreography_manifest.csv`.

## Structural recipe

1. **Open with a concrete screen or claim**: viewer sees the problem immediately.
2. **Prove with UI/source**: show the real page, tool, command, or document.
3. **Translate into diagram**: switch to light-grid/card/arrow scene.
4. **Return to operation**: demonstrate the actual step in software.
5. **Summarize with cards**: compress the takeaway into a clean infographic.

## Must-copy structurally

- Dense but readable visual events.
- UI evidence + diagram explanation alternation.
- Bottom subtitles and on-screen focus marks.
- Programmatic text and vector layers instead of text baked into images.
- Every key sentence has a corresponding visible action.

## Must avoid

- Copying the creator's watermark, brand phrase, exact screenshots, voice, pacing idiosyncrasies, or copyrighted assets.
- One static screenshot talked over for 10 seconds.
- Full-screen bullet slides.
- Decorative icons that do not react to narration.
