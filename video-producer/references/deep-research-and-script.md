# Deep Research And Script

Use this before beat planning when the video needs facts, judgment, critique, product analysis, industry context, or a script that should feel discovered rather than listed.

The goal is depth with low paperwork: enough source lock and narrative structure to prevent shallow commentary, without reviving a heavy pipeline.

## Trigger

Use this path for:

- AI/product/tool analysis or comparison.
- Developer workflow claims such as speed, cost, reliability, generated output, or failure modes.
- News, public events, finance, policy, health, social topics, science, or current facts.
- Any user request containing "深度", "深搜", "深度思考", "探索", "拆解", "分析", "复盘", "真相", "为什么".
- Any draft that reads like "第一、第二、第三" but lacks a central story thread.

Skip or keep minimal for purely fictional motion tests, logo stings, UI-only demos with no factual claims, or user-provided scripts that explicitly ask for no research.

## Minimal Outputs

Use these files when project form exists:

```text
research/source_cards.jsonl
research/claim_ledger.csv
research/factcheck_report.md
script/narrative_thread_map.json
research/thread_ledger.csv
outputs/script.md
```

For very small jobs, `outputs/script.md` may include short sections named `Source Lock`, `Claim Ledger`, and `Narrative Thread` instead of separate files. The constraints are the same.

## Source Lock

Each important source card should record:

- source id, URL/path, title, publisher/author, date captured.
- why it matters to the script.
- usable visual material: screenshot, table, chart, product page, code, terminal output, document, quote, or none.
- risk: current fact, old fact, disputed, interpretation, anecdote, marketing claim, or blocked.

Rules:

- Current prices, laws, public roles, product specs, model names, benchmarks, and news must be checked at production time.
- Prefer official docs/pages, primary sources, product UI, repo/code, changelog, public statements, filings, papers, or direct screenshots.
- Secondary commentary can explain context, but should not be the only support for a factual claim.
- If evidence is weak, mark uncertainty in the script or cut the claim.

## Claim Ledger

Every factual or evaluative line in the script needs one of:

- source id(s),
- explicit uncertainty,
- visible proof generated by the project itself, such as screen recording or terminal output,
- or removal.

Do not infer beyond the screenshot. If a page proves wording, narrate wording. If it proves UI state, narrate UI state. If it does not prove motive, cause, scale, or outcome, do not claim those.

## Narrative Thread

Replace "three points" with a spine:

- **surface:** what viewers think this is about.
- **hidden detail:** a verifiable detail most viewers missed.
- **mechanism:** how the system/workflow/incentive actually works.
- **turn:** the moment that changes interpretation.
- **landing:** what this means, and why the viewer should now see the topic differently.

For multi-point topics, connect points through one master thread instead of listing them. Each section must carry something forward into the next section.

Avoid:

- "首先 / 其次 / 最后" as the main structure.
- empty speculation such as "你想想看" without proof.
- newsy hard cuts such as "此外 / 与此同时" when a causal or thematic bridge is needed.
- clever conclusions that were not earned by evidence.

## Script Rules

- Write spoken Chinese, not report prose.
- One dense idea per sentence; unfamiliar names and numbers get short clauses.
- Evidence appears before or with the line that uses it.
- Every beat must answer: what does the viewer see while hearing this?
- Add landing lines before jumping to a new room: "所以到这里..." / "这意味着..." / "回到刚才那个细节...".
- Keep viewpoint honest: distinguish fact, interpretation, suspicion, and open question.

## Source-To-Visual Binding

Before `beat_plan.json`, tag each important line with its visual proof type:

- `source_screenshot`: official page, doc, table, product UI, public statement.
- `screen_recording`: real workflow, command, browser, app, generated output.
- `code_terminal`: command, file, API call, log, error, diff.
- `chart_table`: count, price, benchmark, timeline, comparison.
- `metaphor_svg`: abstract mechanism that cannot be directly shown.
- `human_context`: real-world impact, user moment, workflow pressure.

Every selected proof must later be directed in `segment_spec.json`: crop, push-in, redbox, cursor, highlight, source label, chart build, code zoom, terminal typing, or timeline reveal.

## Checks

Manual pass criteria:

- The script has a master thread, not a list.
- Important claims are sourced, uncertain, or removed.
- The most interesting detail is specific and verifiable.
- Each section lands before the next section begins.
- The first 20-30 seconds already contains hook, proof, mechanism or turn, and a visual path into the rest of the video.
- Claims and uncertainties are reflected in `research/factcheck_report.md` before beat planning.
